# 품질 테스트와 비기능 테스트
## 1. 기능 적합성 (Functional Stability) 테스트
사용자의 요구사항을 시스템이 얼마나 만족하는지 
  ### 1) 기능 완전성
  사용자가 **요구하는 기능을 얼마나 제공하는지**, 명세기반 테스트
  ### 2) 기능 정확성
  사용자가 **기대하는 수준으로 얼마나 정확하게 동작하는지**, 정확하게 동작하는 기능의 수로 정확성 판단. 명세기반 테스트 or 구조 기반 테스트
  ### 3) 기능 적절성
  사용자의 **목적 달성하는데 도움을 주는 정도**

---

## 2. 성능 효율성 (Performance Efficiency) 테스트
**시스템 운영 프로파일 (Operational profile)** : 실제로 사용자들이 시스템을 사용하는 패턴

성능 효율성 테스트 예시 

#### ex) 벤치마크 테스트 (Benchmark test)
**실제와 상황이 같은 시험 환경에서** 대표적인 비교 대상과 비교 시험을 반복하여 성능을 평가. 시스템이 동작되는 운영 환경 및 작업 부하를 대표해야. (동시에, 동일한 작업부하, 극단적 작업 부하도 테스트, 어느 기간 동안 반영할지도 - **반복성**이 중요)

#### ex) 부하 테스팅 (Load testing)
부하를 계속 증가시키면서 **임계점 (처리량이 더는 증가하지 않거나 CPU사용률, 메모리사용률이 비정상적으로 증가) 찾음** → 병목 지점 찾아 병목현상 제거

#### ex) 스트레스 테스팅 (Stress testing)
**임계점 이상**의 부하를 가함

#### ex) 스파이크 테스팅 (Spike testing)
**짧은 시간에 몰릴 때** 시스템의 반응 측정

#### ex) 내구성 테스팅 (Endurance testing/Soak testing)
**오랜 시간 동안 높은 부하** 가하여 시스템의 반응 측정

  ## 1) 시간 반응성
  ## 2) 자원 효율성
  ## 3) 수용성

#### + 리틀의 법칙 (Little's law)
시스템에 오랜 기간 동안 머물러 있는 고객의 평균 수치는 오랜 시간 동앙네 걸친 평균 시간 도착률 * 시스템에서 고객이 머문 평균 시간이다 ! 

즉
```
동시 사용자 Concurrent User = (처리량 Throughput) * (요청간격 Request Interval)
                           = 처리량 * (응답 시간 + 씽크타임)
                           = (처리량 * 응답시간) + (처리량 * 씽크타임)
                           = Active User + Inactive User 
```
- Ramp up : 가상 사용자를 이용하여 부하를 주는 패턴
- Ramp down : 성능 테스트가 종료된 후에 가상 사용자 수를 줄이는 패턴

## 3. 호환성 (Compatibility) 테스트
다른 시스템과의 **상호연동능력**을 확인하기 위한 테스트, **다른 제품과 공통으로 환경 및 자원 공유**하면서 그 제품에 유해한 영향 미치지않고 올바른 기능 수행하는지. 국방정보시스템에 쓰이는 LISI 모델 (6단계로 이루어져있음)
  ### 1) 공존성
  다른 소프트웨어와 환경 및 자원 공유하면서 요구된 기능 효율적으로 수행하는지. 

  ex) 보안 프로그램들이 메모리와 같은 자원을 사용하여 충돌하거나 자원을 독점적으로 사용할 때 공존성 out
  
  ### 2) 상호운영성
  특별하게 노력을 기울이지 않아도 잘 동작할 수 있는 능력.

## 4. 사용성 (Usability) 테스트
주어진 사용 환경에서 특정 목적을 달성하기 위해서 (효과성, 효율성, 만족도)
  ### 1) 적합인식성
  자신의 필요에 시스템이 적합한지를 인지
  ### 2) 학습 용이성
  소프트웨어 사용법을 배워 명시된 목적을 달성
  ### 3) 운영 용이성
  쉽게 조작하고 제어할 수 있는 속성
  ### 4) 사용자 오류 방지성
  사용자가 오류 X 정도
  ### 5) 사용자 인터페이스 심미성
  사용자에게 만족스러움을 주는 정도
  ### 6) 접근성
  특성이나 능력에 관계없이 시스템 사용할 수 있는 정도

사용성 테스트 예시
#### ex) 휴리스틱 평가 (Heuristic Evaluation)
사용성 원칙 기준으로 체크리스트 이용
  - 적은 인원의 전문가로도 가능 
  - 중요한 사용성 문제 많이 도출
  - 효율적이고 빠르게 수행

#### ex) FGI (Focus Group Interview)
그룹 인터뷰 방법, 시스템 개발 이전에 많이 사용. 공통점이 있는 사용자를 그룹별로 모아 의견을 나누게 함

#### ex) 인지적 워크쓰루
학습 용이성 분석. 실제 사용자 대상으로 과제 달성하도록 해서 행동 분석

결론 : 평가 목적에 따라 평가 방법을 달리하는게 좋다 !

## 5. 신뢰성 (Reliability) 테스트
특정 조건에서 특정 기간 동안 시스템이 요구되는 서비스를 오동작 없이 제공하는 정도.
가용성이나 MTTF (Mean Time To Failure) 등의 척도로 정량화됨. 

**통계적 테스트 (Statistical testing)** 방법 
- 운영 프로파일 사용하여 테스트 케이스 생성
- 클래스의 발생 확률에 따라 테스트 케이스 생성
- 오류가 발생한 시간과 오류발생~다음오류발생때까지 시간 기록

  ### 1) 성숙성
  정상 작동 상태에서 신뢰성 요구를 충족시키는 정도

  ### 2) 가용성
  시스템 또는 구성요소 사용하고자 할 떄 사용 및 접근 가능한 정도

  ### 3) 결함 허용성
  HW나 SW가 결함이 있는데도 의도한 대로 작동하는 정도

  ### 4) 복구성
  중단 또는 장애가 발생한 경우 시스템이 영향 받은 데이터를 복구하고 상태를 재설정할 수 있는 정도

---

## 6. 보안성 (Security) 테스트
시스템이 정보 및 데이터를 보호하는 정도

보안성 테스트 예시
#### 침입 테스트 (Penetration test)
침입자(Hacker)의 관점에서 시스템의 취약성을 찾는 테스트 방법. 가능한 한 많은 시스템 침입 시나리오들 사용

#### 정적 분석
소스 코드 수준에서 코드 규칙 정의.

  ### 1) 기밀성
  ### 2) 무결성
  ### 3) 부인 방지성 (Non-repudiation)
  ### 4) 책임성 (Accountability)
  ### 5) 인증성

+) **CWE (Common Weakness Enumeration)** : 소스 코드에 존재할 수 있는 신뢰성 및 보안성 관련된 코드 규칙 목록 정의

## 7. 유지보수성 (Maintainability) 테스트
**기능 개선 및 추가, 변경된 환경에 적응, 오류 수정, 예상치 못한 장애 예방** 과 같은 변경 요구를 만족시키는 능력

유지보수성 테스트 예시
#### ex) 정적 테스트
실제 소요되는 시간이나 비용을 계산하여 요구사항과 비교되는 방식으로

  ### 1) 모듈성
  ### 2) 재사용성
  ### 3) 분석성
  ### 4) 변경 용이성
  ### 5) 테스트 용이성

테스트하기 위해서 확인해야할 특성들
#### 모듈화 정도 (낮을수록 good)
fan-in과 fan-out이 크면 많은 모듈이 영향을 받기 때문에 정적 분석에서 식별하고 재구성해야 함.

#### 모듈 간 결합도 (낮을수록 good)
잘 정의된 인터페이스만을 통하여 모듈 간 상호작용을 하는 게 바람직

#### 모듈 응집도 (⭐높을수록 good⭐)
응집도가 높으면 의존 관계가 낮기 때문에 !

#### 모듈 복잡도 (낮을수록 good)
ex) 순환 복잡도.
따라서 정적 분석으로 순환 복잡도 높은 모듈 식별하고 리팩토링 + 재구성 통해 복잡도 낮게 만들어야 함

+)
그리고 테스트 용이성에는 **제어 용이성, 관찰 가능성, 단순성, 분할 용이성, 운영 용이성, 안전성, 이해 용이성**이 있음

## 8. 이식성 (Portability)

  ### 1) 적응성
  ### 2) 설치 용이성
  ### 3) 대체 용이성

#### +) 크로스 브라우징 테스트 (Cross Browsing Test)
