테스트 분류 기준 3가지 !
## 분류기준 1) 테스트 레벨
### 1. 컴포넌트 (Component) / 단위 (Unit) 테스트
개별 단위 모듈을 독립적으로 테스트

### 2. 통합 (Integration) 테스트
단위 모듈이 정확하게 통합되었는지, 내부 구성모듈과 이들 간의 **관계**를 정의한 구조 설계 명세서를 바탕으로

### 3. 시스템 (System) 테스트
전체 시스템을 대상으로

### 4. 인수 (Acceptance) 테스트
시스템 테스트와 달리 **고객/사용자의 관점에서 고객이 기대하는 방식으로** 소프트웨어가 동작하는지 확인

#### V 모델
개발 단계 (필요 기대 -> 요구 분석 -> 구조 설계 -> 상세 설계 -> 구현)가 테스트 수준 (컴포넌트 테스트 -> 통합 테스트 -> 시스템 테스트 -> 인수 테스트)와 알파벳 'V' 형태를 이룸

---

## 분류기준 2) 테스트 유형
### 1. 기능 (Functional) 테스트
모든 테스트 수준에서 진행 !!

### 2. 비기능 (Non-functional) 테스트
개별 품질 요구사항 측면의 결함 검출 및 충족 여부 확인. 
시스템 테스트와 인수 테스트 수준에서 진행
ex) ISO 25010 품질 모델의 주특성
- 기능 적합성
- 사용성
- 성능 효율성
- 호환성
- 신뢰성
- 보안성
- 유지보수성
- 이식성

#### 유형 테스트 (Type test)
기능 테스트와 각각의 비기능 테스트

---

## 분류기준 3) 테스트 설계 기법
 **1. 정적 테스트** (대상 실행 X) 
(1장 내용 참고)
### 1-1. 리뷰
전문가 그룹이 수행 ! 순서는 1) 경영진 준비, 2) 리뷰 계획, 3) 리뷰 절차 개요 설명, 4) 작업물 개요 설명, 5) 개별 준비, 6) 그룹 검토, 7) 재작업, 8) 후속 작업
- **관리 리뷰** : 프로젝트 진행 상황 - 인력, 일정, 범위 등
- **기술 리뷰** : 정의된 계획 및 명세 준수
- **인스펙션 (Inspection)** : 문제 식별 + 올바른 해결 검증
- **워크쓰루 (Walkthrough)** : 문제 식별 + 대안 조사, 개선 활동, 학습 기회 제공
- **감사 (Audit)** : 객관적인 표준과 규제에 대한 준수


### 1-2. 정적 분석
산출물의 구조적 속성을 이용하여 자동화된 방식으로 수행
- **코딩 표준**
- **복잡도 측정**
- **자료 흐름 분석** : 프로그램 자료 흐름에 이상 (Anomaly) 존재 여부 분석

---

**2. 동적 테스트** (테스트 대상 실행, 적절한 입력값 즉 테스트 케이스를 결정해야)
(1장 내용 참고)
### 2-1. 명세 기반 테스트
**소스 코드를 참고하지 않고** 테스트 케이스 결정. (=블랙박스 기법) ✏️*적용 대상에 제한이 없음 !* ex) 임의 (Random) 테스트 방법
- 동등 분할
- 분류 트리 기법
- 경곗값 분석
- 신택스 테스트
- 조합 테스트
- 상태 전이 테스트
- 인과 그래핑
- 결정표 테스트
- 시나리오 테스트

### 2-2. 구조 기반 테스트
**구현된 소스 코드 참고**하여 테스트 케이스 결정. (=화이트박스 기법)
- 문장 테스트
- 결정 테스트
- 조건 테스트
- 결정/조건 테스트
- 다중 조건 테스트
- 변형 조건/결정 테스트(MCDC)
- 기본 경로 테스트

### 2-3. 경험 기반 테스트
#### 2-3-1. 오류 추정 (Error Guesssing)
개발자가 범할 수 있는 실수 추정하여 나열, 실수에 따른 결함 노출. 

ex) tan(x)에서 x=90도에 대한 값 특별 처리하지 않았을 실수가 예상됨 -> 테스트 입력에 90도를 포함시킴. 

ex) 그외에도 입력의 크기가 0일 때, 하나의 데이터를 갖고있거나 모두 동일한 데이터를 갖고있거나 모두 정렬이 되어있는 경우
- 필수 입력
- 입력 항목의 길이
- 입력 항목의 형식
- 입력값의 명시적 제약사항 : 명시적으로 정해진 범위 (ex. YY : 00-99, MM : 01-12, DD : 01-31)
- 입력값의 묵시적 제약사항 : 상식적으로 가정되는 범위 (ex. YY : 25 이하의 값; 금년이 2025년이므로)

#### 2-3-2. 탐색적 테스트
테스트 대상에 대한 이해, 테스트 케이스 설계, 테스트 실행 병행. 이해를 바탕으로 **즉석에서 테스트 케이스 결정, 문서화 없이 해당 테스트 바로 수행**. 그리고 테스트 결과 -> 다음 테스트 케이스 결정. 

테스트 차터 (Charter) 간략하게 작성하여 결함 + 추후 해결이 필요한 이슈 기록

그러나 명확한 가이드가 없으면 과도한 시간 낭비 / 동일 기능 중복 + 반복적 테스트 (인력 낭비) / 테스틔 적합성 (커버리지, Coverage) 판단 불가

[오답노트]

✏️*테스트 케이스를 먼저 디자인하지 않음 !* 

✏️*세션 기반 테스팅을 사용하여 테스트를 구조화함* 

✏️*애자일 방법을 사용하는 웹 응용 시스템 (개발 주기가 짧아 세부적인 테스트 명세서를 개발할 수 없기때문에) 의 테스트에 적합함* 

✏️*다른 테스트 기법의 사용을 포함할 수 있음*

## 테스팅 방법
### 리그레션 테스트 (Regression Test)
소프트웨어 **변경 후**에 수행, 변경이 결함을 만들지는 않았는지 or 기존의 요구사항을 충족하는지 검증

-> Retest-all 전략, 선택적 리그레션 테스트 전략, 테스트 최소화 전략, 테스트 우선 순위화 전략

그리고 소프트웨어 변경되면 각 레벨마다 리그레션 테스트 진행 (레벨 테스트 순서대로 컴포넌트 테스트 -> 통합 테스트 -> 시스템 테스트)

---

### 소프트웨어 생명 주기 모델과 테스트
테스트는 소프트웨어 생명 주기 모델의 특성을 고려하여 수행되어야 함. 
- **순차적** 생명 주기 모델이면 **구현이 완료된 시점에서 테스트 1회**
  - 장점 : 필요한 모든 자료가 존재
  - 단점 : 검출된 결함 해결하는데 많은 비용과 시간 소요
  - but V-모델은 순차적 개발 모델이지만 개발 시작과 함께 테스트도 시작됨.

- **반복적이고 점진적**인 모델이면 (ex. 진화형 모델, 애자일 모델) 테스트도 **이터레이션 (Iteration)** + **점진적인 (Incremental) 방식**으로. 

  핵심 부분을 개발한 후 구성 요소와 추가 요구사항을 여러 이터레이션을 통해 개선, 발전, 최종 완성품 개발. 각 이터레이션마다 테스트 수행 계획 작성 -> 수행

  -> 애자일 개발 방법론
    - **테스트 주도 개발(Test-Driven Development, TDD)** 적용. 테스트 케이스 먼저 작성하고 테스트 케이스 통과하는 실제 프로그램 코드 나중에 작성하는 방식.
    - **지속적 통합(CI, Continuous Integration)** 규칙 중요.

---

### 위험 기반 테스트
테스트 대상과 범위를 결정할 때는 테스트 미수행에 따른 위험을 고려해야 함. 따라서 **테스트 제외에 따른 위험 수준이 높은 것은 테스트 대상에 반드시 포함해야**

---

### 모델 기반 테스트
✏️ ️️*테스트 대상의 **동작에 대한 정형적 명세**를 바탕으로 테스트 케이스 및 테스트 절차를 **자동으로 생성**하는 방식으로 수행되는 테스팅* ✏️ 
- 장점 : 자동화, 다양한 유형의 문제를 이른 시점에 식별 -> 안전 필수 (Safety critical) 소프트웨어에 적용
- 단점 : 정형적이고 상세한 테스트 모델을 구축하는 비용이 추가됨
