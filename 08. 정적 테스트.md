# 정적 테스트
# 리뷰(Review) = 정적 테스트
여러 전문가가 모여 프로그램 검토하여 결함 검출. 발생하는 순간 가능한 한 빨리 검출 !
1) 경영진 준비
2) 리뷰 계획
3) 리뷰 절차 개요 설명
4) 작업물 개요 설명
5) 개별 준비
6) 그룹 검토
7) 재작업
8) 후속작업 

## 1) 관리 리뷰
✏️ *관리자가 팀 참여 가능*
진행 상황 모니터, 평가하여 적절히 변경되었는지 확인. 경영적 관점에서
- 설치 계획
- 백업 및 회복 계획
- 안정성 계획
- 재난 계획
- 비상 대책 계획
- 진행 보고서
- 테스트 결과

## 2) 기술 리뷰
유능한 인력이 프로그램의 **기술적 상태 확인**.

대표 엔지니어(Lead engineer)가 주재

✏️ *관리자가 해결해야 할 이슈가 있으면 관리자도 참가 가능*
- 의도된 사용에 적합한지
- 계획, 법규, 표준, 명세 충실히 지키는지
- 변경 명세에 식별된 영역에만 해당 영역이 영향을 미치는지
- 여러 대안 추천, 검토

## 3) 인스펙션 (Inspection)
✏️ *관리자 팀 참여 안됨 !!!!*

가장 형식화된 (절차 O) 리뷰 방식, 동료 검토 (Peer Review)와 같음. 

자원, 시간 낭비 줄일 수 있음 (✏️ *코딩이 시작되면서 투입 인력 소요 ↓, 재작업 ↓* )

개발 초기에 검사해야만 문제 찾아낼 수 있음

인스펙션 참가자의 역할
- 주재자 (Inspection leader/Moderator == Faciliator)
- 작성자 (Author)
- 낭독자 (Reader)
- 기록자 (Recorder)
- 검토자 (Inspector)

인스펙션 과정
1) 리뷰 계획
2) 인스펙션 절차 개요 설명
3) 인스펙션 작업물에 대한 개요 설명
4) 준비
5) 검토 회의
6) 재작업
7) 후속작업

## 4) 워크쓰루 (Walk-through)
✏️ *관리자 팀 참여 안됨 !!!!*
문서를 작성자가 설명, 동료들과 이해도 공유 + 피드백 수렴하는 비형식적인 결함 검출 방법

## 5) 감사 (Audit)
✏️ *관리자 팀 참여 절대안됨 !!!!*

✏️ *목적 : 소프트웨어 제품 및 프로세스가 규제, 표준, 가이드라인, 계획, 절차 준수하고있는지 독립적으로 평가*
